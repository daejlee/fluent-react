그런데 이번장을 읽고 이질감을 전혀 느끼지 못했습니다. 그렇다는건 React는 Next.js의 SSR과 완벽히 호환되는 벤더락인이 심하다는 말이겠지요..??

## RSC란?

RSC는 React Server Components의 약자로, 서버에서 실행되는 React 컴포넌트를 말합니다. RSC는 클라이언트에서 실행되는 React 컴포넌트와 달리, 서버에서 렌더링되어 HTML을 생성하는 역할을 합니다.

우선 서버 컴포넌트나 클라이언트 컴포넌트 둘다 결과는 사용자에게 렌더되는 HTML입니다. 그런데 그 과정이 다음처럼 흘러갑니다.

서버 컴포넌트:
JSX → React 엘리먼트 트리 → 직렬화 → 클라이언트로 전송 → 클라이언트에서 렌더링

클라이언트 컴포넌트:
JSX → React 엘리먼트 트리 (+ 번들화된 로직 코드) → 클라이언트로 전송 → Hydration → 클라이언트에서 렌더링 + 인터랙티브 활성화

### 클라이언트 컴포넌트의 동작 원리

클라이언트 컴포넌트는 `useState`, 이벤트 핸들러 같은 상호작용 로직을 가지고 있습니다. 그럼 이것들이 어떻게 되는걸까요?

1. **서버에서의 역할**: 클라이언트 컴포넌트도 서버에서 **초기 렌더링만** 수행합니다. `useState`나 이벤트 핸들러 같은 상태/로직은 실행되지 않고, 단지 React 엘리먼트 트리로 변환될 뿐입니다.

2. **클라이언트로의 전송**: 서버에서 생성된 React 엘리먼트 트리와 함께 **컴포넌트의 모든 로직 코드(useState, 이벤트 핸들러 등)가 JavaScript 번들로 패키징**되어 클라이언트로 전송됩니다.

3. **Hydration**: 클라이언트에서는 서버가 보낸 초기 HTML을 받고, 번들된 JavaScript 코드를 실행하여 상태와 이벤트 핸들러를 **활성화**합니다. 이 과정을 Hydration이라고 부릅니다.

즉, 클라이언트 컴포넌트는 서버에서 "청사진"을 만들고, 클라이언트에서 "실제 인터랙티브한 버전"으로 살아나는 것입니다.

## 서버액션

이전 장에서 배운 next.js, remix의 서버 액션과 동일한 개념으로, 폼 데이터를 제출하거나 버튼 클릭과 같은 비동기 작업을 처리하기 위해 서버에서 실행되는 함수를 정의할 수 있습니다.

서버 컴포넌트 뿐만 아니라 클라이언트 컴포넌트에서도 서버 액션을 사용할 수 있습니다. 클라이언트 컴포넌트에서 서버 액션을 호출하면, 해당 액션이 서버에서 실행되고 결과가 클라이언트로 반환됩니다.

**Next.js와의 관계:**

- Next.js의 서버 액션은 React의 서버 액션을 그대로 활용합니다.
- 추가로 API Routes라는 별도의 엔드포인트 방식도 제공하는데, 이는 서버 액션과는 다른 접근 방식입니다. API Routes는 HTTP 엔드포인트를 직접 정의하는 반면, 서버 액션은 컴포넌트에 더 통합된 방식입니다.

## turnServerComponentsIntoTreeOfElements

Express 서버에서 RSC를 렌더링할 때, `renderToReadableStream()` 등의 함수 **내부에서 자동으로 호출되는 React 내부 함수**입니다.

이 함수의 역할:

- **JSX 코드를 React 엘리먼트 객체로 변환**
- "서버 컴포넌트를 트리 구조의 자바스크립트 객체로 만든다"는 의미

간단히 말하면:

```
<App /> → { $$typeof: Symbol.for('react.element'), type: App, props: {...} }
```

이 단계를 거친 후에 직렬화해서 클라이언트로 보내는 것입니다. 보통 개발자가 직접 호출할 일은 없고, `renderToReadableStream` 같은 고수준 API가 내부에서 자동 처리합니다.
