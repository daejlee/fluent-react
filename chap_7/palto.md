# 7. 리액트 동시성

## 7.1 동기식 렌더링의 문제

| 동기식 렌더링                                                                                                                                                                           | 동시성 렌더링                                                                                                                                                         |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| - 메인 스레드를 가로막아 사용자 경험 저하<br>- 모든 업데이트를 동일하게 처리, 우선순위 없음 (사용자가 보지 않는 항목도 동일하게 렌더링)<br>- 배치 처리로 일부 완화 가능하지만 한계 존재 | - 작업의 중요도/긴급도에 따라 우선순위 부여<br>(CPU 많이 소모하는 작업 < 사용자 상호작용/애니메이션)<br>- 타임슬라이싱으로 렌더링을 작은 단위로 분할, 점진적 처리<br> |

## 7.2 파이버 다시보기

파이버 재조정자 => **동시성 렌더링**을 가능케 하는 리액트의 핵심 메커니즘

- 렌더링 프로세스를 **파이버**라는 작은 작업 단위로 분할해 처리
- 리액트는 파이버를 활용해 렌더링 작업을 일시중단/재개/우선순위를 이용해 업데이트 지연/예약

## 7.3 업데이트 예약과 지연

애플리케이션의 응답성을 유지하는데 중요한 개념 -> **파이버에선 스케줄러와 여러 API를 통해 구현**

`사용자 입력 (높은 우선순위) ←→ 새 메시지 수신 렌더링 (낮은 우선순위)` 이 같은 상황에서 동기식 렌더링 방식이면 사용자의 텍스트 입력이 막히는 문제가 발생

=> **`useTransition`** 을 이용하여 우선순위를 낮출 수 있다. 리액트의 예약시스템과 관련된 훅으로, 낮은 우선순위 레인으로 위치시켜 렌더를 늦게 예약시키는 훅이다.

```js
const ChatApp = () => {
  const [messages, setMessages] = useState([]);
  const [isPending, startTransition] = useTransition();

  useEffect(() => {
    const socket = new WebSocket("wss://your-websocket-server.com");

    socket.onmessage = (event) => {
      startTransition(() => {
        // 낮은 우선순위로 예약 → 사용자 입력을 가로막지 않음
        setMessages((prev) => [...prev, event.data]);
      });
    };

    return () => socket.close();
  }, []);

  return (
    <div>
      <MessageList messages={messages} />
      <MessageInput onSubmit={sendMessage} />
    </div>
  );
};
```

`sendMessage` > `setMessages`

## 7.4 스케줄러

`스케줄러`는 타이밍 관련 유틸리티를 제공하는 독립형 패키지로 **파이버재조정자와는 별개로 동작**한다.

마이크로태스크를 예약하여 메인 스레드의 제어를 관리하고 원활한 실행을 보장한다.

### 마이크로태스크

**매크로태스크 큐**

이벤트 처리, setTimeout, setInterval, 입출력 작업 등이 저장되는 곳.  
한번에 하나씩 처리되며, 현재 작업이 완료된 후에만 다음 작업 진행

**마이크로태스크 큐**

promise, observe, mutationObserver같은 연산에서 발생.  
마이크로태스크는 현재 작업이 끝나고 태스크큐에서 매크로태스크를 실행하기 전에 먼저 실행됨.  
마이크로태스크가 전부 비워진 후에야 다음 매크로태스크 실행

```
[매크로태스크 실행]
    ↓
[마이크로태스크 큐 전부 비움] ← 브라우저가 렌더링이나 다음 이벤트 처리를 하기 전에 루트 스케줄을 먼저 처리
    ↓
[브라우저 렌더링]
    ↓
[다음 매크로태스크 실행]
```

**기아 상태**  
다음 매크로 태스크로 넘어가기 전, 마이크로태스크가 계속 대기열에 추가될 때 계속 작업 대기열이 처리되지 않는 상태

### ensureRootIsScheduled 함수

리액트 루트에 업데이트가 발생할 때마다 호출되는 함수

- 루트가 스케줄에 포함되도록 보장
- 루트 스케줄을 처리할 수 있도록 대기 중인 마이크로 태스크가 존재하는지 확인

(대부분의 실제 예약 로직은 scheduleTaskForRootDuringMicrotask) 함수가 실행될 때 수행

```js
export function ensureRootIsScheduled(root: FiberRoot): void {
  // 루트를 스케줄에 추가
  if (root === lastScheduledRoot || root.next !== null) {
    // 이미 스케줄링된 루트 → 넘어감
  } else {
    // 루트가 스케줄에 포함되도록 보장
    if (lastScheduledRoot === null) {
      firstScheduledRoot = lastScheduledRoot = root; // 첫 루트
    } else {
      lastScheduledRoot.next = root;  // 목록 끝에 추가
      lastScheduledRoot = root;
    }
  }

  mightHavePendingSyncWork = true; // 동기 작업 가능성 표시

  // 마이크로태스크 예약 (아직 예약 안 된 경우에만)
  if (!didScheduleMicrotask) {
    didScheduleMicrotask = true;
    scheduleImmediateTask(processRootScheduleInMicrotask);
  }
}
```

```js
if (nextLane === Sync) {
  queueMicrotask(processNextLane); // 긴급: 마이크로태스크로 즉시 처리
} else {
  Scheduler.scheduleCallback(processNextLane); // 비긴급: 나중에 처리
}
```

다음 레인이 Sync(가장 높은 우선순위)라면 마이크로태스크큐에 추가, 아니라면 콜백(매크로태스크큐)으로 예약해 나중에 처리

## 7.5 렌더 레인

작업의 렌더링과 우선순위 관리를 효율화함.

- 레인은 우선순위 수준을 나타내는 단위로, 리액트가 렌더링 주기에서 처리할 수 있는 작업을 의미.
- React 18에서 도입 (이전엔 만료 시간이 있는 예약 메커니즘)
- setState를 호출하면 해당 업데이트가 레인으로 보내짐. 어느 레인으로 가는지는 호출된 콘텍스트에 따라 달라짐

### 주요 레인의 종류

#### SyncHydrationLane

하이드레이션 도중 사용자가 리액트 앱을 클릭하면 클릭 이벤트가 이 레인으로 들어감

#### SyncLane

사용자가 리액트 앱을 클릭하면 클릭 이벤트가 이 레인으로 들어감

#### InputContinuousHydrationLane

하이드레이션 도중 발생하는 호버, 스크롤을 비롯한 연속적인 이벤트

#### DefaultLane

모든 네트워크 업데이트, 타이머, 우선순위가 파악되지 않는 초기 렌더링

#### TransitionLanes

하이드레이션 후 startTransition에서 발생하는 모든 트랜지션

#### RetryLanes

Suspense에서 발생하는 모든 재시도

### 리액트 동작 방식

#### 1. 업데이트 수집

마지막 렌더링 이후에 예약된 모든 업데이트 수집하여 우선순위에 따라 레인에 할당

#### 2. 레인 처리

우선순위가 가장 높은 레인부터 각 레인에 있는 업데이트 처리. 같은 레인의 업데이트는 일괄 처리

#### 3. 커밋 단계

모든 업데이트 처리 후 커밋 단계로 진입 해 변경사항을 DOM에 적용  
커밋이 완료되면 레인 제거

#### 4. 반복

### 레인 할당 작동 방식

**1. 업데이트의 콘텍스트 확인**  
업데이트가 사용자 상호작용인지, 서버 응답인지, 백그라운드 작업인지 파악. 콘텍스트가 우선순위 결정의 핵심

**2. 콘텍스트에 따라 우선순위 추정**  
사용자 입력 → 높은 우선순위 / 백그라운드 작업 → 낮은 우선순위로 자동 추정

**3. 우선순위 재정의 확인**  
개발자가 `useTransition` 또는 `useDeferredValue`로 명시적으로 우선순위를 지정했다면 추정값 대신 해당 값을 사용

**4. 레인에 업데이트 할당**  
결정된 우선순위에 따라 알맞은 레인에 배정. 내부적으로 **비트마스크(bitmask)** 를 사용해 여러 레인을 효율적으로 관리

### 레인 처리 순서 (채팅 앱 예시)

#### 1. ImmediatePriority

메시지 입력에 대한 업데이트 처리

#### 2. UserBlockingPriority

입력 상태 표시에 대한 업데이트 처리

#### 3. NormalPriority

메시지 목록의 업데이트를 처리
