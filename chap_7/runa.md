# 7 - 리액트 동시성

## 1. 동기식 렌더링의 한계

동기식 렌더링은 리액트가 업데이트를 시작하면 완료될 때까지 메인 스레드를 점유하는 방식이다. 이로 인해 다음과 같은 문제가 발생한다.

### 1) 사용자 경험 저하 및 메인 스레드 차단

컴포넌트가 많고 업데이트가 빈번한 복잡한 애플리케이션에서 UI 반응이 느려지거나 응답을 하지 않는 현상이 생긴다. 리액트가 무거운 렌더링 작업을 수행하는 동안 사용자의 클릭이나 타이핑 같은 긴급한 입력이 무시되기 때문이다.

### 2) 우선순위 개념의 부재

동기식 렌더링은 모든 업데이트를 동일하게 취급한다.

- 사용자가 볼 수 없는 탭이나 모달 뒤의 콘텐츠 등 중요하지 않은 업데이트가 사용자의 즉각적인 피드백이 필요한 작업을 가로막을 수 있다.
- 일괄 처리로 작업 횟수를 줄일 수는 있지만, 어떤 작업이 더 긴급한지 판단하여 먼저 처리하는 기능은 부족하다.

## 2. 파이버 재조정자

### 1) 작업 단위의 분할

파이버는 렌더링 프로세스를 파이버 노드라고 하는 작은 작업 단위로 쪼개어 관리한다. 이를 통해 리액트는 작업을 일시 중단하거나 다시 시작할 수 있는 제어권을 갖게 된다. 이는 고성능 애플리케이션에서 메인 스레드를 효율적으로 사용하여 프레임 드랍을 방지하는 데 도움을 준다.

### 2) 스냅샷과 데이터 불일치 이슈

파이버가 작업을 쪼개어 처리하기 때문에, 리액트는 내부적으로 여러 버전의 UI 트리를 준비할 수 있다.

- 각 렌더링 시점마다 컴포넌트는 그 순간의 데이터를 사진 찍듯 캡쳐(Snapshot)한다.
- 전역 변수처럼 리액트 외부에서 빠르게 변하는 값을 참조할 경우, 렌더링 시점이 미세하게 갈리면서 서로 다른 버전의 UI가 다른 값을 보여주는 일시적 불일치 현상이 발생할 수 있다.

## 3. 스케줄러

스케줄러는 setTimeout이나 MessageChannel 같은 브라우저 API를 활용하여 작업의 실행 순서를 관리한다. 내부적으로는 렌더 레인과 비트마스크를 사용하여 업데이트의 긴급도를 계산하고, 우선순위가 섞인 복잡한 상황에서도 효율적으로 실행 순서를 결정한다.

### 1) ensureRootIsScheduled

리액트 루트에 업데이트가 발생할 때마다 호출되는 핵심 함수이며 다음 두 가지를 보장한다.

1. **루트 등록:** 해당 루트가 처리해야 할 목록(루트 스케줄)에 포함되어 있는지 확인하고, 중복되지 않게 관리한다.
2. **마이크로태스크 확인:** 스케줄을 처리하기 위해 대기 중인 마이크로태스크가 존재하는지 확인하여, 현재 실행 중인 스크립트가 끝나자마자 즉시 렌더링이 시작되도록 한다.

### 2) 자바스크립트 이벤트 루프와 큐

- **매크로태스크 큐:** `setTimeout`, `setInterval`, 입출력 작업 등이 담기며, 한 번에 하나씩 처리된다.
- **마이크로태스크 큐:** `Promise`, `MutationObserver` 등이 담긴다. 현재 실행 중인 작업이 끝나면 다음 매크로태스크로 넘어가기 전에 이 대기열의 모든 작업을 즉시 실행한다.
- 리액트는 마이크로태스크를 활용해 UI 업데이트를 매우 높은 우선순위로 처리한다. 이를 통해 렌더링이나 이벤트 처리 같은 다른 매크로태스크가 실행되기 전에 최신 UI를 준비한다.

### 3) 레인에 따른 예약 로직

스케줄러는 작업이 속한 렌더 레인의 긴급도에 따라 다른 예약 방식을 선택한다.

```tsx
// 스케줄러 내부의 핵심 동작 원리
if (nextLane === SyncLane) {
  // 1. 동기 레인: 마이크로태스크 큐에 직접 추가 (가장 빠름)
  queueMicrotask(processNextLane);
} else {
  // 2. 비동기 레인: 스케줄러 콜백으로 예약 (중단/지연 가능)
  Scheduler.scheduleCallback(processNextLane);
}
```

- 우선순위가 높은 마이크로태스크가 계속해서 새로운 마이크로태스크를 추가할 경우, 매크로태스크가 영원히 실행되지 못하는 **기아 상태**에 빠질 수 있다.

## 4. 렌더 레인

- 리액트가 렌더링 주기에서 처리할 수 있는 작업의 우선순위 수준을 의미한다.
- 내부적으로 32비트 정수를 사용하여 총 31개의 레인을 관리한다. 비트 연산을 통해 여러 레인을 동시에 그룹화하거나 특정 레인의 포함 여부를 매우 빠르게 계산할 수 있다.

### 1) 주요 레인 종류

| **레인 이름**           | **발생 콘텍스트**                                    | **우선순위** |
| ----------------------- | ---------------------------------------------------- | ------------ |
| **SyncLane**            | 사용자 클릭, 입력 등 즉각적인 피드백이 필요한 이벤트 | 가장 높음    |
| **InputContinuousLane** | 호버(Hover), 스크롤 등 연속적으로 발생하는 이벤트    | 높음         |
| **DefaultLane**         | 네트워크 요청(Fetch), `setTimeout`, 초기 렌더링      | 중간         |
| **TransitionLanes**     | `startTransition`으로 감싸진 상태 업데이트           | 낮음         |
| **RetryLanes**          | Suspense 내에서 실패한 작업을 다시 시도할 때         | 낮음         |

### 2) 작동 방식

리액트는 업데이트가 발생한 **콘텍스트**를 평가하여 적절한 레인을 할당하고 처리한다.

- 업데이트 처리 단계
  1. **업데이트 수집:** 마지막 렌더링 이후 예약된 모든 업데이트를 수집해 레인에 할당한다.
  2. **레인 처리:** 우선순위가 가장 높은 레인부터 순차적으로 처리하며, 같은 레인의 작업은 일괄 처리한다.
  3. **커밋 단계:** 모든 계산이 끝나면 변경 사항을 실제 DOM에 반영하고 Effect를 실행한다.
  4. **반복:** 높은 우선순위 작업이 들어오면 낮은 순위 작업을 중단하고 다시 렌더링 주기를 시작한다.
- 우선순위 결정 요인
  - **콘텍스트 확인:** 사용자 입력인지, 서버 응답인지, 혹은 타이머에 의한 것인지 판단한다.
  - **명시적 재정의:** `useTransition`이나 `useDeferredValue`를 사용하여 특정 업데이트의 우선순위를 낮출 수 있다.

## 5. useTransition

**상태 업데이트**의 우선순위를 낮춰 UI 차단을 방지하는 훅이다.

### 1) 구성 요소

- **isPending:** 트랜지션이 진행 중인지 알려주는 상태값이다. 리액트는 `startTransition` 호출 시 즉시 이를 true로 설정하는 동기 업데이트를 예약한다.
- **startTransition:** 지연시키고자 하는 상태 업데이트 로직을 감싸는 함수다. 이 내부의 작업은 `Transition Lane`에 배치된다.

### 2) 핵심 활용 사례 → 페이지 탐색

- **상황:** 단일 페이지 애플리케이션에서 새 페이지로 이동할 때, 다음 페이지가 무거운 데이터를 페칭해야 하는 경우 UI가 멈출 수 있음
- **동작:** `startTransition`으로 페이지 변경 상태를 감싸면, 리액트는 백그라운드에서 새 페이지를 준비한다. 그동안 `isPending`은 true가 되어 개발자가 로딩 스피너 등을 자유롭게 보여줄 수 있으며, 사용자 입력 같은 긴급한 작업은 여전히 빠르게 처리된다.
- **Suspense와의 결합:** 트랜지션 내부에서 일시 중지가 발생하면, 리액트는 데이터와 렌더링이 완료될 때까지 `isPending` 상태를 유지하며 구버전의 UI를 보여준다.

## 6. useDeferredValue

특정 UI 업데이트를 나중으로 미루어 애플리케이션의 응답성을 유지하는 훅이다.

### 1) 작동 메커니즘: Stale-While-Revalidate

- 업데이트가 발생하면 리액트는 새 값이 준비될 때까지 이전 값을 화면에 계속 유지한다.
- 새 값으로 업데이트할 시점을 리액트가 스스로 판단하여, 여러 번의 중간 렌더링 없이 한 번에 최종 값으로 업데이트한다.
- 지연된 값을 처리하는 도중 새로운 입력이 들어오면, 리액트는 진행 중이던 렌더링을 즉시 중단하고 새 입력에 먼저 대응한다.

```tsx
function useDeferredValue(value) {
  const [newValue, setNewValue] = useState(value);
  useEffect(() => {
    // 값을 트랜지션으로 감싸서 업데이트함 (지연 처리)
    startTransition(() => {
      setNewValue(value);
    });
  }, [value]);
  return newValue;
}
```

### 2) 디바운싱/스로틀링과의 차이점

임의로 시간을 설정해야 하는 전통적인 기법과 달리, `useDeferredValue`는 **장치 성능에 최적화된 지연**을 제공한다.

| **구분**        | **디바운싱 / 스로틀링**             | **useDeferredValue**                        |
| --------------- | ----------------------------------- | ------------------------------------------- |
| **지연 시간**   | 고정적 (예: 1초 대기)               | **동적** (기기 성능에 따라 가변적)          |
| **중단 가능**   | 불가 (일단 실행되면 멈출 수 없음)   | **가능** (새 입력 시 렌더링 중단 후 재시작) |
| **사용자 경험** | 성능 좋은 기기에서도 고정 지연 발생 | 성능 좋은 기기에서는 거의 즉시 반영         |

### 3) 활용 및 최적화

1. useMemo와 memo
   - 리액트는 `deferredValue`가 변하기 전, 최신 `value`를 반영하기 위해 컴포넌트를 먼저 한 번 실행한다. 이때 무거운 계산 로직이 `useMemo`로 보호되지 않으면, 지연 효과를 보기도 전에 메인 스레드가 차단된다.
   - 하위 컴포넌트에 지연된 값을 props로 넘길 때, 해당 컴포넌트를 `memo`로 감싸야만 불필요한 리렌더링을 방지하고 실제 지연된 시점에만 UI를 업데이트할 수 있다.
2. 주요 사례
   - 대규모 데이터 목록의 필터링 및 검색 결과 표시
   - 복잡한 차트나 시각화 애니메이션의 렌더링 지연
   - 사용자 입력과 데이터 처리의 분리

### 4) 주의사항

- 업데이트를 지연시키면 사용자는 일시적으로 과거의 데이터를 보게 된다. 정보의 최신성이 절대적으로 중요한 경우에는 부적절하다.
- 사용자가 즉각적인 반응을 기대하는 Input 자체의 값에는 사용해서는 안 된다.

## 7. 티어링

애플리케이션이 의존하는 상태가 렌더링 도중에 변경되어 시각적 불일치 버그를 유발할 수 있다.

### 1) 발생 원인

동시성 렌더링은 작업을 쪼개서 수행한다. 만약 렌더링 비용이 매우 큰 컴포넌트(`ExpensiveComponent`)가 여러 개 있다면, 리액트가 이들을 하나씩 처리하는 사이에 외부 상태가 변할 수 있다.

- **ExpensiveComponent:** 의도적으로 `while` 루프 등을 사용하여 렌더링 시간을 100ms 이상 점유하도록 만든 컴포넌트
- **불일치 메커니즘**
  1. 첫 번째 `ExpensiveComponent`가 렌더링될 때 전역 변수 `count`는 **100이**다.
  2. 이 컴포넌트가 무거운 연산을 하는 100ms 동안, 외부의 `setInterval`에 의해 `count`가 **101**로 증가한다.
  3. 리액트가 두 번째 `ExpensiveComponent`를 렌더링할 때는 이미 값이 **101**로 변해 있다.
  4. 결과적으로 한 화면에 100과 101이 동시에 나타나는 티어링이 발생한다.

### 2) useSyncExternalStore

리액트 외부의 상태를 안전하게 읽기 위해 도입된 훅이며 렌더링 중 값의 일관성을 강제로 보장한다.

```tsx
// 1. 외부 저장소(Store) 정의
const store = {
  count: 0,
  subscribe(callback) {
    // 값이 변할 때마다 리액트에게 알림을 보냄
    const id = setInterval(() => {
      this.count++;
      callback();
    }, 1);
    return () => clearInterval(id);
  },
  getSnapshot() {
    return this.count; // 현재 시점의 값을 반환
  },
};

// 2. 컴포넌트에서 사용
const ExpensiveComponent = () => {
  // 전역 변수를 직접 쓰지 않고 훅을 통해 읽음
  const consistentCount = useSyncExternalStore(store.subscribe, store.getSnapshot);

  // 무거운 연산 시뮬레이션
  const now = performance.now();
  while (performance.now() - now < 100) {}

  return <div>카운트: {consistentCount}</div>;
};
```

### 3) 해결 원리

- `getSnapshot`은 리액트가 한 주기의 렌더링을 시작하면, 그 주기가 끝날 때까지 모든 컴포넌트에게 동일한 값을 반환하도록 리액트 내부에서 관리된다.
- 외부 값이 변해 `subscribe`의 콜백이 호출되면, 리액트는 이를 지연 가능한 작업이 아닌 동기적 작업으로 처리하여 렌더링 도중 값이 변할 틈을 주지 않는다.
