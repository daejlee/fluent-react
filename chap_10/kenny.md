프레임워크, 서버 컴포넌트의 역할과 기능에 대한 이해는 리액트의 대체제를 알아보는 과정에서도 중요한 배경 지식이 된다. 리액트에서 다룬 많은 원칙과 트레이드오프가 각 라이브러리의 생태계에도 적용된다.

# 10.1 Vue.js

UI 작성에 널리 사용되는 JS 프레임워크다. 뷰의 반응형 모델은 객체 속성의 읽기와 쓰기를 가로챈다.

```tsx
function reactive(obj) {
  return new Proxy(obj, {
    get(target, key) {
      track(target, key);
      return target[key];
    },
    set(target, key, value) {
      target[key] = value;
      trigger(target, key);
    },
  });
}

function ref(value) {
  const refObject = {
    get value() {
      track(refObject, 'value');
      return value;
    },
    set value(newValue) {
      value = newValue;
      trigger(refObject, 'value');
    },
  };
  return refObject;
}
```

프록시를 활용하고, reactive 함수는 객체를 전달받아 해당 객체의 프록시를 반환하고, get, set 작업을 가로챈다. 반면 ref 함수는 객체 내에서 값을 캡슐화하고 해당 값에 대해 반응형 get, set 작업을 제공한다.

## 10.1.1 시그널

다른 프레임워크들도 refs와 유사한 반응형 특성을 ‘signal’이라는 용어로 도입한다. 값에 접근할 때는 의존성을 추적하고, 값이 변경되면 사이드이펙을 일으키는 값 컨테이너다.

오늘날 시그널은 세분화된 구독을 통해 업데이트를 수행하는 렌더링 모델과 함께 자주 의논되곤 한다. 가상 DOM을 사용하는 뷰는 유사한 최적화를 위해 컴파일러를 사용한다.

## 10.1.2 단순성

뷰의 강점은 단순성으로, HTML의 script 태그에 뷰 라이브러리를 포함, 컴포넌트 작성을 시작하면 된다.

# 10.2 Angular

광범위한 FE 문제에 대한 자체적인 방식을 제공하고, 고유 방식을 강력하게 권장하는 프레임워크다. 가상 DOM과 재조정 대신 변경 감지라는 시스템을 사용한다. 앵귤러에는 모든 컴포넌트에 zone.js라는 라이브러리를 사용해 뷰에 변경 사항이 있는지 확인하는 변경 감지기가 있다.

## 10.2.1 변경 감지

앵귤러는 컴포넌트를 위에서 아래로 살펴보며 변경 사항을 찾는다. 이 변경 감지 메커니즘을 주기적으로 실행해 변경 사항이 앱의 뷰에 반영되게 한다. 이는 수동으로, 아니면 비동기 이벤트로 트리거할 수 있다.

## 10.2.2 시그널

앵귤러는 자체적으로 구현한 반응형 특성을 도입해 변화를 겪고 있다.

```tsx
const count = signal(0);

count(); // 값에 접근
count.set(1); // 새로운 값 설정
count.update((v) => v + 1); // 기존 값을 기반으로 업데이트

// 동일한 객체의 내부 값 수정
const state = signal({ count: 0 });
state.mutate((o) => {
  o.count++;
});
```

이 게터 기반 API를 뷰의 refs와 비교하면 장단이 드러난다.

- ()는 .value보다 코드가 적지만 업데이트 부분에선 더 번잡함
- 값에 접근하려면 항상 ()가 필요하다. 이를 통해 어디서나 일관된 방식으로 값에 접근할 수 있다.

앵귤러의 강제적인 특성은 일관성과 구조를 부여하지만, 유연성과 학습 곡선에서 불리하다.

# 10.3 Svelte

스벨트는 UI 구축에 접근하는 근본적으로 새로운 방식을 제시한다. 선언적 컴포넌트를 명령형 코드(imperative code)로 변환하는 컴파일러를 써 DOM을 정교하게 업데이트한다. 그로 인해 더 적은 코드로 고성능 앱을 작성한다.

```tsx
<script>
let count = 0;

function increment() {
	count += 1;
}
</script>

<div>{count}</div>
<button on:click={increment}>
	클릭하세요
</button>
```

스벨트는 반응형 동작이 세터 함수를 호출하거나 특별한 API를 사용해 DOM을 업데이트할 필요 없이 변수에 할당하기만 하면 나머지는 스벨트가 처리한다.

스벨트의 컴파일러 방식은 장점이 있는데, 가상 DOM을 안써서 런타임 성능이 빨라진다. 대신 DOM을 직접 업데이트 하는 코드를 생성한다.

단점으로는, 동적 컴포넌트 타입같은 가상 DOM 기반의 프레임워크에서 제공하는 일부 동적 기능은 표현이 장황하다. 스벨트 생태계는 규모가 작고 미숙하다.

# 10.4 Solid

솔리드는 UI를 구축하기 위한 선언적 JS 라이브러리다. 컴포넌트 모델 기반이라는 점에서 리액트와 비슷하지만, 반응형 특성에 기반하는게 다르다. 가상 DOM 대신 세분화된 반응형 시스템을 써서 의존성을 자동으로 추적하고 DOM을 직접 업데이트한다.

```tsx
import { createSignal } from 'solid-js';

function Component() {
  const [count, setCount] = createSignal(0);

  return (
    <>
      <div>{count()}</div>
      <button onClick={() => setCount(count() + 1)}>증가</button>
    </>
  );
}
```

createSignal은 리액트의 useState와 유사하다. 다만 count가 현잿값을 반환하고 반응형 콘텍스트에 대한 의존성을 암시적으로 등록하는 함수라는 것이다.

리액트의 경우 setCount가 호출되면 컴포넌트 블록 내부의 모든 로직을 포함해 다시 호출된다. count 값 자체는 반응형이 아니다.

솔리드의 경우 컴포넌트 함수가 다시 호출되지 않지만, count 값 자체는 반응형으로 동작해 setCount마다 변경된다. 이를 세밀한 반응성(fine-grained reactivity)이라고 하며, 리액트의 거친 반응성(coarse-grained reactivity)과는 정반대다.

# 10.5 Qwik

퀵은 페이지 로딩을 최적화하고 사용자 상호 작용과 응답성을 우선시하도록 설계된 프레임워크다. 웹 페이지를 컴포넌트의 집합으로 보고, 네트워크를 통해 독립적으로 읽어 들이고 필요에 따라 상호 작용할 수 있는 대상으로 취급한다. 이는 페이지 초기 로드 시간을 크게 단축한다.

퀵의 필수 기능은 재개 가능성이다.

# 10.6 자주 사용하는 패턴

위 프레임워크들 모두 차이가 있지만 공동의 목적을 반영하는 공통점도 몇 가지 있다.

## 10.6.1 컴포넌트 기반 아키텍처

컴포넌트 기반 아키텍처를 공유한다. 컴포넌트는 자체 상태와 로직을 캡슐화한다.

## 10.6.2 선언적 구문

선언적 구문으로 UI를 정의한다. 선언적 접근 방식에서는 개발자가 특정 상태에 대해 UI의 모양을 설정하면 프레임워크가 해당 상태에 맞게 UI를 업데이트하는 작업을 처리한다.

## 10.6.3 업데이트

앱 상태의 업데이트에 응답하고 그에 따라 UI를 변경하는 메커니즘을 제공한다. 리액트와 뷰는 가상 DOM 비교 알고리즘을 사용하고, 스벨트는 컴포넌트를 명령형 코드로 컴파일해 DOM을 직접 업데이트한다. 앵귤러는 zone.js와 옵저버블 기반의 변경 감지 메커니즘을 사용한다.

## 10.6.4 수명 주기 메서드

컴포넌트가 생성될 때, DOM에서 제거될 때 등 컴포넌트 수명 주기의 각 단계에 맞는 함수를 호출한다. 이를 수명 주기 메서드, 훅이라고 부른다. 이를 통해 사이드이펙을 실행하고 리소스를 정리, 프롭의 변경에 따라 업데이트를 수행한다.

## 10.6.5 생태계와 도구

위 프레임워크들 모두 최신 자바스크립트 기능과 도구를 지원한다. TS도 지원한다. 대부분 디버깅이나 프로파일링에 도움이 되는 개발자 도구가 존재한다.

모두 독자적이지만 여러 목표를 공유한다. 이런 공통된 기능과 개념은 웹 개발이 모듈적이고 선언적이며 반응형 패러다임으로 진화한다는 증거다.

# 10.7 반응형이 아닌 리액트

리액트는 정확히 말하면 전통적인 의미의 “반응형”은 아니다.

리액트가 '반응형'이 아니라고 불리는 이유는 데이터가 변했을 때 바뀐 부분만 자동으로 감지해 업데이트하는 게 아니라, 컴포넌트 함수 전체를 다시 실행하기 때문.

전통적인 반응형 프레임워크(스벨트, 솔리드 등)는 데이터와 UI를 실시간으로 연결해 값이 변하면 그 자리에서 즉시 반영하지만, 리액트는 '상태가 바뀌었으니 화면을 다시 그려라’라고 명시적으로 명령(setState)해야만 작동하게 된다.

결국 리액트는 자동화된 반응성 대신, v=f(s)라는 공식을 바탕으로 가상 DOM을 통해 변경 사항을 비교하고 반영하는 방식을 취했다. 이는 시스템이 알아서 움직이는 것보다 개발자가 UI 변화를 더 명확하게 예측하고 제어할 수 있게 해준다.

# 10.8 리액트의 미래

리액트 팀은 다른 프레임워크처럼 '시그널' 방식을 도입하는 대신, 리액트 컴파일러(React Compiler)를 통해 성능 문제를 해결하려 한다. (시그널에 대해서는 흥미가 없다고 한다)

핵심은 개발자가 수동으로 `memo`나 `useMemo`를 써서 최적화하던 번거로운 작업을 컴파일러가 코드를 분석해 자동으로 처리해 주는 것. 이를 통해 리액트 고유의 프로그래밍 모델을 유지하면서도, 시그널에 뒤처지지 않는 고성능 UI를 구현하는 것을 목표로 한다.

결국 "UI는 상태의 함수"라는 철학은 지키되, 기계가 알아서 최적화된 코드를 만들어내어 개발자는 오직 기능 구현에만 집중할 수 있는 방향으로 보고있다.
