# 10. 리액트 대체제

이번 장은 한 줄로 요약하면 이겁니다.
"결국 다 컴포넌트로 UI를 만들지만, 업데이트 철학이 다르다."

리액트를 오래 쓰다 보면 상태 바꾸고 리렌더 되는 흐름이 너무 당연한데,
다른 프레임워크를 보면 "아 이쪽은 아예 다른 사고방식이네"가 바로 느껴집니다.

## 뷰

### 상태관리

Vue 3의 핵심은 `Proxy` 기반 반응성입니다.
객체 읽을 때(track), 쓸 때(trigger)를 가로채서 필요한 곳만 다시 반영합니다.

여기서 포인트 하나만 더 잡으면,
`reactive`는 "객체"에 강하고, `ref`는 "원시값(숫자/문자열 등)"까지 감싸기 좋습니다.
그래서 실무에서는 둘을 섞어 쓰는 경우가 많습니다.

```javascript
function reactive(target) {
  return new Proxy(target, {
    get(target, key) {
      track(target, key);
      return target[key];
    },
    set(target, key, value) {
      target[key] = value;
      trigger(target, key);
      return true;
    },
  });
}

function ref(value) {
  const refObject = {
    get value() {
      track(refObject, "value");
      return value;
    },
    set value(newValue) {
      value = newValue;
      trigger(refObject, "value");
    },
  };
  return refObject;
}
```

제가 느낀 Vue의 장점은 "설명하기 쉬운 반응성"입니다.
객체는 `reactive`, 단일 값은 `ref`로 나눠서 이해하면 거의 다 풀립니다.

### 시그널

사실 Vue의 `ref`가 이미 시그널에 가까운 개념입니다.
값 컨테이너를 들고 있고, 값을 읽는 시점에 의존성을 기록하고,
값이 바뀌면 구독자(이펙트)를 깨우는 구조라서요.

> 시그널은 쉽게 말해 "값 + 자동 알림"입니다.
> 엑셀에서 A1 값을 바꾸면 A1을 참조한 셀들이 자동으로 다시 계산되는 것처럼,
> 시그널도 값을 읽은 곳을 기억해뒀다가 값이 바뀌면 그 부분만 다시 갱신합니다.
> React처럼 컴포넌트 전체를 다시 실행하는 감각보다,
> "이 값 쓰는 지점만 콕 집어서 업데이트"에 더 가까운 패러다임입니다.

다만 Vue는 보통 템플릿/컴포넌트 단위 개발 경험이 강하고,
시그널 라이브러리들은 "값 단위 API"를 더 전면에 내세운다는 차이가 있습니다.

## 앵귤러

### 변경 감지

Angular는 오랫동안 `zone.js` 기반 변경 감지로 갔습니다.
비동기 작업이 생기면 감지 사이클을 돌고, 컴포넌트 트리를 검사해서 업데이트합니다.

요약하면 "광범위하게 확인해서 안전하게 맞춘다"에 가까운 전략입니다.
최근에는 시그널과 함께 zone 의존도를 줄이는 방향(zoneless)도 같이 발전 중입니다.

장점은 "프레임워크가 다 해준다"에 있고,
단점은 구조를 강하게 따라야 해서 러닝커브가 꽤 높다는 점입니다.

### 시그널

최근 Angular는 시그널을 공식 도입해서 반응성을 더 세밀하게 가져가고 있습니다.

```typescript
const count = signal(0);

count();
count.set(1);
count.update((v) => v + 1);
```

개인적으로 `count()` 형태는 처음엔 어색한데,
적응되면 "반응형 값 접근"이 눈에 딱 보여서 나쁘지 않았습니다.

## 스벨트

### 룬

Svelte 5에서는 룬(`$state`, `$derived`, `$effect`)으로 반응성 규칙을 명시합니다.
예전보다 "어디가 반응형인지"가 코드에서 더 분명해졌습니다.

#### 런타임 반응성

스벨트는 기본적으로 컴파일러 중심입니다.
가상 DOM diff에 의존하기보다, 빌드 시점에 DOM 업데이트 코드를 뽑아냅니다.
그래서 런타임 오버헤드가 작은 편이고 체감 성능도 좋습니다.

React 관점에서 보면 "렌더 결과를 비교하는 엔진"을 런타임에 두기보다,
"어떤 값이 바뀌면 어디를 업데이트할지"를 미리 컴파일해두는 느낌입니다.

#### 시그널

스벨트가 API 이름을 꼭 signal로 부르진 않아도,
본질은 "값 단위 의존성 추적 + 필요한 부분만 갱신"으로 같습니다.

## 솔리드

Solid는 시그널 철학을 제일 직설적으로 보여주는 쪽입니다.
`createSignal`로 상태를 만들고, 값을 읽은 정확한 위치만 업데이트합니다.

```tsx
const [count, setCount] = createSignal(0);
```

리액트처럼 컴포넌트 함수 전체를 다시 실행하는 모델이 아니라,
필요한 계산/DOM만 재실행하는 fine-grained reactivity에 가깝습니다.
성능 관점에서 "왜 시그널 얘기가 계속 나오는지"를 이해하기 좋은 프레임워크였습니다.

React의 `useState`와 모양은 비슷해도,
Solid의 getter 함수 호출(`count()`) 자체가 의존성 추적 포인트라는 점이 핵심 차이입니다.

## 퀵

Qwik의 키워드는 resumability(재개 가능성)입니다.
초기 로딩 때 JS를 최대한 늦추고, 사용자 상호작용이 일어난 시점에 필요한 코드만 깨워서 실행합니다.

쉽게 말해 기존 Hydration 비용을 줄이는 수준을 넘어,
"가능하면 Hydration 자체를 최소화/회피"하려는 접근입니다.
"인터랙션이 없는 코드까지 왜 처음부터 다 실행해야 하지?"라는 질문에 정면으로 답하는 느낌입니다.

## 반응형이 아닌 리액트

엄밀히 말하면 "리액트는 반응형이 아니다"라기보다,
"시그널 기반의 세밀한 자동 반응형과는 모델이 다르다"가 더 정확합니다.

리액트도 상태/props 변화에 반응해 UI를 업데이트하므로 큰 범주의 반응형 UI 라이브러리는 맞습니다.
다만 업데이트 단위가 값 그 자체라기보다 컴포넌트 렌더링 단위에 더 가깝습니다.

리액트는 전통적인 의미의 자동 반응형(값 바뀌면 그 값의 소비 지점만 즉시 갱신)과는 결이 다르고,
`setState`를 통해 "다시 렌더 사이클"을 돌리고,
그 결과를 비교(diff)해서 DOM에 반영하는 모델이죠.

즉 리액트는
"상태를 직접 추적해서 자동 갱신"보다는
"UI = f(state)를 다시 계산하고 일관성 있게 반영"에 더 가깝습니다.

React만 익숙한 입장에서 기억하면 좋은 비교는 아래 3개입니다.

1. React: 상태 변경 -> 컴포넌트 재실행 -> 결과 비교 후 반영
2. Signal 계열: 상태 변경 -> 그 값을 읽던 지점만 직접 갱신
3. Svelte/Qwik: 런타임 일을 컴파일/직렬화 전략으로 최대한 앞당기거나 쪼갬

그래서 "리액트가 뒤처졌다"라기보다는,
동일 목표(빠른 UI)를 서로 다른 설계 철학으로 푸는 중이라고 보는게 가장 정확합니다.

결론적으로 이 장에서 제가 정리한 포인트는 이겁니다.

1. 리액트 대체제들은 대부분 시그널/세밀한 반응성 쪽으로 가고 있다.
2. 리액트는 그 대신 컴파일러 최적화와 기존 모델 유지 쪽을 선택했다.
3. 어느 쪽이 무조건 정답이라기보다, 팀이 원하는 DX/성능/학습비용 균형의 문제다.
